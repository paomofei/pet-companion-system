# 多线程协作使用手册

## 目标

本手册用于规范 `Pet_Sys` 项目的多线程协作方式，确保同一份产品规划、数据库设计、前后端实施计划和接口契约可以被多个工作线程稳定复用，减少返工和联调冲突。

当前项目默认采用 4 类线程：

- 基线线程
- 前端线程
- 后端线程
- 联调线程

## 基本原则

1. `API接口契约.md` 是前后端联调的唯一接口依据。
2. `伴读宠物产品规划_完整版.md`、`产品架构设计.md`、`伴读宠物数据库设计_修订版.md`、`前端实施计划.md`、`后端实施计划.md` 是项目实现依据，不重复生成替代版本。
3. 线程之间只按职责分工，不按个人理解随意扩展字段、接口或流程。
4. 公共规则优先在基线线程处理，前端线程和后端线程不擅自修改公共约束。
5. 联调线程只做集成验证和最小收口，不承担大范围重构。

## 线程定义

### 基线线程

职责：

- 维护项目基线文档和公共约束。
- 维护跨前后端共享规则。
- 处理接口契约冲突、公共类型冲突和联调规则。
- 负责最终合并和联调收口。

禁止事项：

- 不重复做产品拆解。
- 不在没有必要时承担大段前端实现。
- 不在没有必要时承担大段后端实现。

### 前端线程

职责：

- 实现页面、组件、交互、状态管理和接口调用层。
- 严格按接口契约对接请求和返回结构。
- 在后端尚未完成时使用符合契约的 mock 数据。

禁止事项：

- 不私自修改 `API接口契约.md`。
- 不私自增加接口字段。
- 不处理后端 service、数据库和持久层逻辑。

### 后端线程

职责：

- 实现接口、service、鉴权、数据模型和持久层。
- 严格按数据库设计文档和接口契约实现。
- 保证请求校验、错误处理、返回结构稳定。

禁止事项：

- 不私自修改 `API接口契约.md`。
- 不私自调整数据库语义以适配前端临时实现。
- 不承担前端页面实现。

### 联调线程

职责：

- 验证前端与后端是否按契约联通。
- 检查请求、响应、错误处理、空态和边界情况。
- 修复最小必要的联调阻塞问题。
- 输出联调结果和剩余阻塞点。

禁止事项：

- 不大规模重构前后端。
- 不在联调阶段扩展新功能。
- 不跳过契约直接“按跑通为准”修改返回结构。

## 必读文档

每个线程开始工作前，必须先读取以下文件中的相关内容：

- `AGENTS.md`
- `伴读宠物产品规划_完整版.md`
- `产品架构设计.md`
- `伴读宠物数据库设计_修订版.md`
- `前端实施计划.md`
- `后端实施计划.md`
- `API接口契约.md`
- `package.json`

推荐读取范围：

- 基线线程：全部文件
- 前端线程：重点读取产品规划、架构设计、前端实施计划、接口契约
- 后端线程：重点读取产品规划、架构设计、数据库设计、后端实施计划、接口契约
- 联调线程：全部文件 + 当前联调范围对应代码

## 本轮基线同步（2026-03-13）

本轮已由基线线程统一解决的公共问题如下，前端线程和后端线程必须先检查对应基线文档，再按各自实施计划顺序执行：

1. XP 规则统一为“升级后保留余数”。
2. `POST /wishes` 的公共入参统一为 `rarity`，后端内部映射 `weight`。
3. `POST /wishes/draw` 改为请求体携带 `clientRequestId`。
4. 相同 `clientRequestId` 重复提交时，后端返回第一次抽奖结果，`code=0`。
5. 点击金色🎁后，前端先发起请求，请求处理中播放翻牌动效，结果返回后再揭晓。
6. 抽奖仅在线；离线时允许查看/新增/编辑心愿，但不允许执行抽奖。
7. 抽奖请求超时后，前端使用同一个 `clientRequestId` 自动重试 1 次。
8. 后端新增 `wish_draw_requests` 幂等表，作为抽奖重复请求重放依据。

前端线程执行要求：

- 先检查 `伴读宠物产品规划_完整版.md`、`产品架构设计.md`、`前端实施计划.md`、`API接口契约.md`。
- 按 `前端实施计划.md` 的阶段顺序推进，不跳阶段、不自行改契约。
- 抽奖链路必须按“在线校验 → 生成 clientRequestId → 发请求 → 播放翻牌动效 → 结果揭晓”实现。
- 若发现接口字段、错误码或状态流与文档不一致，停止自行扩展，回报基线线程。

后端线程执行要求：

- 先检查 `伴读宠物产品规划_完整版.md`、`产品架构设计.md`、`伴读宠物数据库设计_修订版.md`、`后端实施计划.md`、`API接口契约.md`。
- 按 `后端实施计划.md` 的阶段顺序推进，不跳阶段、不自行改契约。
- 抽奖链路必须实现 `clientRequestId` 幂等、`wish_draw_requests` 持久化、重复请求返回第一次结果。
- 返回结构以 `API接口契约.md` 为准，不按数据库字段直接外露。

本轮监督重点：

- 前端是否严格按实施计划阶段推进，而不是先跳到联调或私自改接口。
- 后端是否先补齐数据模型和契约，再进入 service 与事务实现。
- 双方是否都以 `API接口契约.md` 为唯一联调依据。
- 遇到阻塞是否先回报基线线程，而不是各自改公共文档。

## 线程创建顺序

推荐顺序如下：

1. 在主线程 `Pet_Sys` 中确认基线文件已存在。
2. 在主线程中确认 `AGENTS.md`、`package.json`、`.nvmrc` 等协作文件已就位。
3. 从主线程派生前端工作树。
4. 从主线程派生后端工作树。
5. 当前后端分别完成阶段性实现后，再开联调线程或回主线程做联调。

## 推荐线程命名

- `Pet_Sys`：基线版
- `前端开发`：前端线程
- `后端开发`：后端线程
- `联调验证`：联调线程

## 共享文件边界

以下文件默认只允许在基线线程修改：

- `AGENTS.md`
- `API接口契约.md`
- `package.json`
- `.nvmrc`
- `.github/workflows/*`
- 共享类型定义
- 跨前后端公共配置

前端线程和后端线程如发现这些文件需要调整，先汇报问题，再由基线线程统一处理。

## 标准协作流程

### 阶段 1：基线确认

1. 在主线程确认产品规划、架构设计、数据库设计、实施计划和接口契约已齐备。
2. 确认本轮开发范围和基线版本。
3. 确认协作文件和 Node 版本约束已生效。

### 阶段 2：前后端并行开发

1. 前端线程按前端实施计划实现页面和调用层。
2. 后端线程按后端实施计划实现接口和数据层。
3. 双方都必须严格遵守接口契约。
4. 如发现契约问题，回到基线线程统一修正。

### 阶段 3：联调验证

1. 按优先级验证登录、核心主页面加载、关键操作、错误处理和持久化。
2. 优先修复契约偏差和真实联通问题。
3. 只做最小必要修改，不扩展功能。

### 阶段 4：主线程收口

1. 先合并后端，再合并前端，或按阻塞关系决定合并顺序。
2. 在主线程统一执行联调和最终检查。
3. 输出当前可用结论、剩余问题和下一步。

## 推荐验证顺序

日常开发：

1. `npm run env:check-node`
2. `npm run check:fast`

提交前或准备联调前：

1. `npm run check:full`

说明：

- 如果项目尚未初始化真实工程脚手架，`lint`、`typecheck`、`test`、`build` 可能还是占位脚本。
- 占位脚本需要在真实工程初始化后替换。

## 联调优先级

推荐按以下顺序联调：

1. 登录与鉴权链路
2. 首页或主工作台数据加载
3. 目标、任务、成长等核心业务操作
4. 异常状态和错误返回
5. 刷新后的状态一致性
6. 边界情况和空态

## 常见误区

### 误区 1：前端线程和后端线程同时改接口契约

后果：

- 最终契约不可控
- 联调时难以判断谁偏离了基线

正确做法：

- 所有契约修订回基线线程处理

### 误区 2：前端为了赶进度自行发明字段

后果：

- mock 和真实接口不一致
- 联调阶段返工

正确做法：

- 严格按契约 mock
- 缺字段时汇报给基线线程

### 误区 3：后端按数据库设计直接输出字段，不看接口契约

后果：

- 返回结构与页面消费模型不一致

正确做法：

- 数据库设计是持久层依据
- 接口契约才是返回结构依据

### 误区 4：联调线程顺手扩功能

后果：

- 联调阶段失焦
- 问题来源混乱

正确做法：

- 联调线程只做最小收口

## 输出要求

每个线程在完成一次任务后，建议输出：

1. 本次处理范围
2. 已完成部分
3. 未完成部分
4. 是否存在契约冲突或阻塞点
5. 下一步建议

本轮要求前端线程和后端线程额外补充：

6. 当前完成到实施计划的第几阶段/哪几个接口
7. 是否已重新检查本轮基线同步 8 条结论
8. 下一步是否会触及公共约束；若会，先暂停并回报基线线程

## 一句话使用规则

主线程管规则和契约，前端线程只做前端，后端线程只做后端，联调线程只做集成验证和最小收口。
