# 伴读宠物系统 - 产品规划与数据库设计一致性审查报告

> 审查对象：`伴读宠物产品规划.docx` × `伴读宠物数据库设计.docx`
>
> 审查目的：逐项交叉比对两份文档，找出不一致、矛盾、遗漏，给出明确修改建议。

---

## 🔴 严重问题（影响核心闭环，必须修改）

### 问题 1：`energy_balance` 归属矛盾 — 到底挂在 users 表还是 pets 表？

- **产品规划 §3.1**：左侧面板展示"能量点余额"，属于**用户资产**
- **数据库 §5.2.2**（ER 说明部分）：明确把 `energy_balance` 和 `ticket_balance` 写在 **pets 表**（原文："宠物表 (pets)：作为左侧面板的数据源，承载了所有的'资产'和'养成进度'"）
- **数据库 §5.3.2**（核心表结构部分）：又把 `energy_balance` 写在了 **users 表**（原文："用户与资产表 (users)：承载用户的基本信息、当前宠物状态以及核心虚拟资产余额"）
- **数据库 §5.4.1**（数据流转部分）：打卡后操作的是 `users` 表的能量余额
- **数据库 §5.2.3**（级联操作部分）：取消打卡时操作的又是 `UPDATE pets SET energy_balance`

> **结论**：同一份数据库文档内部自相矛盾。`energy_balance` 和 `ticket_balance` 在 §5.2.2 说归 pets，在 §5.3.2 说归 users，在不同的数据流转描述中两者混用。

**修改建议**：统一挂在 **users 表**。理由：
1. 能量是"用户"的资产，不是"宠物"的资产（用户用能量喂宠物，是主人喂宠物的逻辑）
2. V1 一个用户只有一只宠物，但未来可能多宠物，资产应该属于用户级别
3. 统一后 §5.2.2 中 pets 表的描述需删除 `energy_balance` 和 `ticket_balance` 字段
4. §5.2.3 中取消打卡的 SQL 也需改为 `UPDATE users SET energy_balance = ...`

---

### 问题 2：任务状态字段命名和值不一致

- **产品规划 §2.2.1**：任务三态 = `待完成 (Pending)` / `已完成 (Completed)` / `已漏做 (Missed)`
- **数据库 §5.2.3**：使用 `is_completed = false/true` 布尔值（原文 `UPDATE tasks SET is_completed = false`）
- **数据库 §5.3.2**（核心表结构）：任务表只提到了 `status` 字段
- **数据库 §5.4.1**（流转描述）：打卡后 `status` 更新为 `completed`

> **结论**：布尔值 `is_completed` 无法表达三态（Pending/Completed/Missed）。文档内有时用 `is_completed`，有时用 `status`。

**修改建议**：
1. 统一使用 `status TINYINT`，值为 `0=待完成, 1=已完成, 2=已漏做`（符合 §5.1.4 规范中"严禁使用 ENUM，统一用 TINYINT"）
2. 删除所有 `is_completed` 的引用，统一改为 `status`
3. §5.2.3 中的 SQL 更正为 `UPDATE tasks SET status = 0, completed_at = NULL`

---

### 问题 3：许愿池机制 — "抽奖" vs "直接兑换"冲突

- **产品规划 §1.2 业务目标**：核心闭环写的是"宠物养成 → **抽奖激励**"
- **产品规划 §3.1 面板**：展示"**抽奖次数**余额"
- **产品规划 §3.2.3**：描述了完整的**加权随机抽奖算法**（权重、随机数、奖池消耗逻辑）
- **产品规划 §4.6（许愿池视图）**：UI 设计却是**直接兑换**模式——心愿有明确售价（⚡200），用户用能量直接购买，没有任何随机性
- **数据库 §5.3.2**：`wishes` 表的 `cost_energy` 也是直接定价模式
- **数据库 §5.4.4**（升级奖励）：升级发放的是 `ticket_balance`（抽奖券），但许愿池 UI 用的是能量兑换，不需要抽奖券

> **结论**：产品内部存在两套互相矛盾的激励机制——一套是抽奖（§3.2.3），一套是直接兑换（§4.6）。UI 框线图展示的均为直接兑换模式。而宠物升级发放的"抽奖券"在直接兑换模式下完全没有使用场景。

**修改建议**（二选一）：

- **方案 A：保留直接兑换（推荐，与 UI 一致）**
  - 删除 §3.2.3 抽奖权重算法部分
  - 将核心闭环描述从"抽奖激励"改为"许愿兑换"
  - 将面板中的"抽奖次数余额"改为其他用途，或删除 `ticket_balance` 字段
  - 宠物升级奖励需要重新定义（建议改为直接奖励固定能量，如升级一次奖励 50 ⚡）

- **方案 B：保留抽奖机制**
  - 重设计许愿池 UI，加入抽奖轮盘/转盘概念
  - wishes 表加 `weight` 字段
  - 兑换逻辑改为消耗抽奖券（而非能量）触发随机抽取

## 🟡 中等问题（影响局部功能，建议修改）

### 问题 4：任务表日期字段命名不一致

- **数据库 §5.2.2**（ER 说明）：使用 `due_date`
- **数据库 §5.3.3**（索引部分）：使用 `target_date`
- **数据库 §5.5.3**（慢查询优化）：也使用 `target_date`

> 同一张表的同一个字段，在文档不同章节用了两个名字。

**修改建议**：统一为 `target_date`（语义更明确，表示"任务归属日期"而非"截止日期"）。

---

### 问题 5：顺延逻辑缺少数据库字段支撑

- **产品规划 §2.2.1**：顺延时"系统会在指定日期生成一条全新的 Task Instance（标记为 `is_delayed_copy = true`）"
- **数据库 §5.3.2**：tasks 表结构中**未包含** `is_delayed_copy` 字段

**修改建议**：在 tasks 表中添加 `is_delayed_copy BOOLEAN DEFAULT false` 字段，或添加 `source_task_id INT NULL` 字段来关联原始任务。

---

### 问题 6：软删除机制不统一 — `deleted_at` vs `is_deleted`

- **数据库 §5.1.5**（规范章节）：明确要求使用 `deleted_at` 时间戳字段（原文："将该记录的 `deleted_at` 字段更新为当前时间"）
- **数据库 §5.3.1**（架构说明）：又写成 `is_deleted`（原文："软删除（is_deleted）"）
- **数据库 §5.2.3**（用户注销）：又写成 `is_deleted = 1`

> 同一文档中两种软删除方案混用。

**修改建议**：统一为 `deleted_at TIMESTAMP NULL`（符合 §5.1.5 的规范要求，且 `deleted_at` 比 `is_deleted` 信息量更大——既知道是否删除，又知道何时删除）。

---

### 问题 7：`onboarding_option` 字段缺失

- **产品规划 §4.1.5**：引导页有"选项A 一键开启"和"选项B 自定义"两种模式，很可能需要记录用户选了哪个
- **数据库**：users 表中未设计 `onboarding_option` 字段

**修改建议**：在 users 表添加 `onboarding_option TINYINT DEFAULT 0 COMMENT '0=一键开启, 1=自定义'`，用于埋点分析引导页转化。

---

### 问题 8：成长页所需的"连续打卡"数据缺少辅助存储

- **产品规划 §4.5.2**：成长页需要展示"🔥连续打卡天数"、"⚡累计获取能量"、"✅累计完成任务"
- **数据库**：没有在任何表中存储这些统计数据

> 虽然可以在查询时实时计算，但连续打卡天数的计算逻辑复杂（需遍历历史日期判断连续性），如果用户使用半年以上，每次打开成长页都全量计算会影响性能。

**修改建议**：在 users 表添加缓存字段：
- `current_streak INT DEFAULT 0`（当前连续打卡天数，每次打卡时动态维护）
- `max_streak INT DEFAULT 0`（历史最长连续天数）
- `total_energy_earned INT DEFAULT 0`（累计获取能量）
- `total_tasks_done INT DEFAULT 0`（累计完成任务数）

每次打卡/取消打卡时增量维护这些字段，避免全量聚合查询。

---

### 问题 9：成长页的"三大核心页签"描述与实际 UI 不一致

- **产品规划 §3.1**：右侧三大核心页签为 `目标页(Goals)` + `任务页(Tasks)` + `成长页(Growth)`，其中成长页描述为"包含宠物互动区与**愿望抽奖区**"
- **产品规划 §4.5**（成长页实际设计）：成长页实际设计为"高光数据 + 7日图表 + 勋章墙"，宠物互动在左侧面板，愿望在许愿池

> §3.1 的概要描述与 §4.5 的详细设计不一致。

**修改建议**：将 §3.1 中成长页的描述从"包含宠物互动区与愿望抽奖区"更正为"包含高光数据看板、近期趋势图表与荣誉勋章墙"。

---

## 🟢 轻度问题（不影响功能，建议优化）

### 问题 10：章节编号紊乱

- 产品规划文档的章节编号严重混乱：1.1 出现了两次（文档概述和修订记录），1.2 出现了两次（背景和产品结构），1.3 出现了两次。更深层编号如 2.1.2、2.1.3 等被反复使用。

**修改建议**：重新统一编排章节号。

---

### 问题 11：数据库文档中"5.3.2 核心表" 与 "5.2.2 核心表"内容重复

- §5.2.2 和 §5.3.2 都在描述核心数据表结构，但说法不同（例如资产归属矛盾就源于此）

**修改建议**：合并为一处，消除冲突源。

---

### 问题 12：宠物表缺少 `pet_name` 字段

- **产品规划 §4.1**：引导页需要输入"宠物昵称"
- **数据库 §5.2.2**：pets 表中没有 `name` 字段

> 宠物名字如果存在 users 表中（如 `pet_name`），则 users 和 pets 之间存在职责模糊。

**修改建议**：在 pets 表中添加 `name VARCHAR(20)` 字段来存储宠物昵称。或者确认放在 users 表并明确标注。

---

### 问题 13：勋章/成就系统缺少数据库表

- **产品规划 §4.5.3**：明确要求勋章解锁机制、勋章系列分类、解锁条件、全屏庆祝
- **数据库**：没有任何与勋章/成就相关的表设计

**修改建议**：新增两张表：
- `badges`（勋章字典表）：id, code, name, description, icon, category, threshold
- `user_badges`（用户勋章表）：id, user_id, badge_id, unlocked_at

---

### 问题 14：wishes 表缺少 `weight` 权重字段

> 如果最终保留抽奖机制（问题 3），则 wishes 表需要 `weight INT` 字段支撑加权随机算法。如果走直接兑换模式则不需要。

---

### 问题 15：`sync_status` 字段在离线场景下的遗漏

- **产品规划 §4.1.6**：离线场景需要 `sync_status = pending` 标记
- **数据库**：各表中均未设计 `sync_status` 字段

**修改建议**：如果 V1 支持离线同步，需在 tasks、goals、wishes 等表中添加 `sync_status TINYINT DEFAULT 1 COMMENT '0=待同步 1=已同步'`。如果 V1 暂不做离线同步可忽略。
