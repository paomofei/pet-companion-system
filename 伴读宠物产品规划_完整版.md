> **本文档是最终权威版本**，合并了原始 `伴读宠物产品规划.docx` 全部内容与 V4 修订。

## 1. 文档概述

### 1.1 产品背景与目标

**产品背景**：当前市面上的儿童任务清单工具多以"打卡打勾"为主，工具属性过强，缺乏趣味性与情感陪伴；而纯粹的电子宠物游戏又容易让孩子沉迷，脱离现实生活。家长在帮助孩子养成习惯时，往往面临"短期缺乏动力，长期难以坚持"的痛点。

**产品定位**：一款融合"虚拟宠物陪伴"与"现实愿望激励"的儿童习惯养成工具。

**业务目标 (V1)**：
- 【V4修订】跑通"**设定目标 → 任务完成 → 获得能量⚡ → 陪伴宠物 → XP满格升级 → 🎁亮起 → 点击即抽**"的核心业务闭环。
- 验证"双轨激励机制"（短期情绪价值+长期现实奖励）对儿童打卡留存率的提升效果。
- 次日留存率目标设定为 45%，首周任务打卡完成率目标设定为 60%。

## 2. 产品结构与业务流程

### 2.1 核心业务流程图

说明：本流程展示了从家长规划到孩子执行，再到最终获得现实激励的完整闭环。

### 2.2 状态机说明

#### 2.2.1 任务实例状态机
- **待完成 (Pending)**：系统基于规则生成的默认状态。
- **已完成 (Completed)**：用户点击打卡后的终态。
- **已漏做 (Missed)**：超过自然日 23:59:59 未打卡，系统自动翻转为该状态。
- **顺延衍生逻辑**：当任务处于"已漏做"或"待完成(昨日)"，用户点击【顺延】时，原任务状态不变（保留漏做记录），系统会在指定日期生成一条全新的 Task Instance（标记为 `is_delayed_copy = true`）。

#### 2.2.2 宠物升级与XP状态机
- **经验积累**：当前 XP < max_xp 时的常态。
- **【V4修订】XP 满格触发升级+抽奖机会**：当 `当前XP + 获得XP >= max_xp` 时：
  - `level += 1`
  - `current_xp = (当前XP + 获得XP) % max_xp`（**保留余数**）
  - `pending_draw = true`（pets 表布尔值，非计数器）
  - 🎁礼物盒从灰色变为金色亮起
  - 举例：当前 LV.2 (90/100 XP)，获得 15 XP → 升至 LV.3，**XP 为 5/100**。
- **用户点击🎁** → 直接触发抽奖动效 → 抽奖完成后 `pending_draw = false`，🎁变回灰色。
- **注意**：若升级时 `pending_draw` 已为 true（用户尚未点击🎁），不额外累加，保持 true。此场景极少发生。

## 3. 全局公共规则说明

### 3.1 页面框架结构 (全局Layout)

为强化"伙伴陪伴"的概念，App 采用 **左侧常驻面板 + 右侧功能页签** 的全局布局结构（平板端横屏为左右分栏，手机端竖屏为上下分栏或悬浮抽屉）。

**左侧/顶部常驻区 (宠物面板)**：
- 【V3修订】展示宠物Live2D/动效立绘、当前等级、XP进度条+**🎁礼物盒**、**仅⚡能量点余额**。
- 伴随场景化气泡对白（如早晨提示："早上好，今天有3个任务哦！"）。

**右侧/底部内容区 (三大核心页签)**：
- 目标页 (Goals)：重规划，管理大目标与任务规则。
- 任务页 (Tasks)：重执行，按日历维度展示每日待办卡片。
- 【V3修订】成长页 (Growth)：重激励，包含**高光数据看板（🔥连续打卡/⚡累计能量/✅累计任务）、近7日趋势图表与荣誉勋章墙**。

### 3.2 核心数值规则

#### 3.2.1 任务难度与能量点产出规则
V1版本采用**"锚定难度"**的极简规则。

| 难度 | 产出 |
|------|------|
| 🥉 小任务 | +10 ⚡ |
| 🥈 中任务 | +20 ⚡ |
| 🥇 大任务 | +50 ⚡ |
| 自定义 | 1-100 ⚡ |

#### 3.2.2 宠物互动与XP转化规则
能量点只能用于宠物互动，不同互动行为的"能量-XP"转化效率不同，鼓励孩子攒能量进行高级互动。

| 道具 | 消耗⚡ | 获得XP |
|------|--------|--------|
| 🍪 小饼干 | 10 | 2 |
| 🍎 鲜水果 | 20 | 5 |
| 🎾 玩球 | 30 | 8 |
| 🍖 大肉肉 | 50 | 15 |
| 🥏 飞盘 | 80 | 25 |

#### 3.2.3 愿望抽奖权重算法
愿望池采用**"加权随机抽取"**算法，确保低价值愿望高频出现，高价值愿望低频出现，控制家长现实兑现的成本。

【V3修订】抽奖档位（替代原"能量定价"）：

| 稀有度 | 权重(weight) | 抽中概率近似 | 适用场景示例 |
|--------|-------------|-------------|-------------|
| ⭐ 小愿望 | 50 | 高频出现 | 看30分钟动画片、选一个冰淇淋口味 |
| ⭐⭐ 中愿望 | 20 | 中频出现 | 周末去公园、买一本新漫画书 |
| ⭐⭐⭐ 大愿望 | 5 | 低频出现 | 买一个新玩具、全家去游乐园 |

**抽取逻辑**：
1. 计算当前奖池总权重 `Total Weight = Σ(各愿望权重)`。
2. 生成 `[0, Total Weight)` 之间的随机浮点数 R。
3. 遍历奖池，累加权重值，当累加值 > R 时，即命中该愿望。
4. 奖池消耗逻辑：抽中的愿望状态变更为"已实现"，自动移出待抽取奖池。若奖池为空，则阻断抽奖操作。

### 3.3 异常处理与防刷机制

#### 3.3.1 客户端防连点/防刷机制
- **前端防抖**：所有"打卡完成"按钮、"宠物互动"按钮、"抽奖"按钮，必须加入 500ms 的防抖处理。
- **动效阻断**：点击打卡后，播放 1~2 秒的能量收集动画，动画播放期间该卡片不可再次操作。
- **每日上限预警**：系统不硬性限制每日获取能量上限，但需在埋点监控中关注异常用户。

#### 3.3.2 空状态规范

| 场景 | 空状态引导文案 |
|------|--------------|
| 目标列表为空 | "还没有设定大目标哦，和爸爸妈妈一起定个小目标吧！" |
| 当日任务为空 | "今天还没有安排任何任务哦~给宠物赚点能量口粮吧！" |
| 许愿池为空 | "还没有许愿哦，先许几个愿望吧！" |
| 勋章全锁定 | "完成任务解锁第一枚勋章吧！" |

#### 3.3.3 数据同步与离线策略
- **离线可用**：用户在无网络状态下，依然可以进行打卡、互动，以及查看/新增/编辑心愿。
- **抽奖仅在线**：`🎁` 抽奖依赖服务端权威随机与幂等校验；离线时点击金色🎁，提示"网络不可用，恢复后再抽奖"。
- **乐观更新**：打卡、互动、心愿管理操作后，UI 立即响应，后台将操作指令写入"待同步队列"；抽奖不进入离线队列。
- **网络恢复重试**：静默按序执行队列。若遇版本冲突，以客户端最新时间戳为准。

## 4. 页面与功能需求详细说明

### 4.1 初始化引导页 (/onboarding)

#### 4.1.1 页面定位与业务目标
- **页面定位**：新用户首次打开 App 时的破冰与基础信息初始化页面。
- **业务目标**：
  - 建立情感连接：通过动态宠物形象和拟人化对白，让孩子立刻感受到"陪伴感"。
  - 降低认知负荷：仅要求填写最核心的2个昵称。
  - 极速进入核心闭环：通过"默认目标组"，一键进入任务页。

#### 4.1.2 前置条件与入口
- **前置条件**：App 首次安装启动，且本地无有效用户数据。
- **设备标识**：客户端首次安装时生成并本地持久化一个应用级 UUID，通过 Header `X-Device-Id` 参与静默识别；V1 一台设备仅绑定一个用户。
- **进入路径**：启动闪屏页结束后，自动路由拦截跳转至 `/onboarding`。
- **退出路径**：成功提交后页面销毁，从路由历史栈中清除。

#### 4.1.3 页面 UI 布局
页面采用竖屏单页设计，从上至下分为四个核心区块：
1. **欢迎头部**：大字号标题"欢迎来到学习伙伴计划"，副标题"领养你的专属学习小助手"。
2. **宠物展示区**：占屏约 1/3 高度，展示默认宠物的动态立绘 + 气泡对话框。
3. **信息录入区**：包含"孩子昵称"与"宠物昵称"两个输入框。
4. **开局选择与操作区**：两种初始化模式的单选卡片 + 底部【开启旅程】按钮。

#### 4.1.4 核心功能与交互
- **昵称输入**：孩子昵称（必填1-10字）+ 宠物昵称（必填1-10字）+ 实时敏感词校验。
- **开局选项**：
  - 选项A "一键开启日常习惯"（推荐）：静默创建默认目标+3个任务 → 跳转 `/today?tab=tasks`
  - 选项B "自定义专属目标"：跳转 `/today?tab=goals` → 自动弹起新增目标弹窗
- 【V3修订】需记录选择项：写入 `users.onboarding_option`（0=A, 1=B），用于漏斗分析。

#### 4.1.5 流转逻辑
- 选项A → `/today?tab=tasks`，宠物气泡："[昵称]，这是我们今天的新任务，快去完成赚取能量吧！"
- 选项B → `/today?tab=goals`，自动弹起新增目标弹窗，宠物气泡："[昵称]，你想定一个什么样的大目标呢？"

#### 4.1.6 异常处理
- **离线**：V1 "本地优先"策略，断网也 100% 成功写入本地。
- **敏感词**：Toast "哎呀，这个名字不太合适，换一个吧~"，按钮恢复激活。
- **杀进程**：再次打开重新进入 /onboarding，清空重填。

#### 4.2.3 【V4修订】XP 进度条 + 🎁礼物盒 二态

| 状态 | 条件 | 视觉 | 点击行为 |
|------|------|------|---------|
| 灰色 | `pending_draw = false` | 灰色静态图标 | Toast "多陪伴宠物升级就能抽奖哦" |
| 金色亮起 | `pending_draw = true` | 金色 + 光晕脉冲CSS动画 | **直接触发抽奖动效**（不打开复杂 Modal） |

**升级→抽奖完整时序**：
- T=0-200ms：互动事务返回 `leveledUp:true` → 进度条白光扫过 → XP 重置为余数
- T=200ms：等级数字放大1.2x变色
- T=400ms：纸屑爆发
- T=600ms：🎁从灰变金 + scale(0→1.3→1.0) 弹跳
- T=800ms：宠物气泡 P0："升级啦！点击🎁看看有什么惊喜！"
- **用户点击🎁**：
  - 若离线 → Toast "网络不可用，恢复后再抽奖"
  - 若在线 → 生成 `clientRequestId` 并立即请求抽奖接口 → 请求处理中播放 1-2s 翻牌动效 → 返回后揭晓中奖心愿 + Confetti → 宠物 P1 气泡 → 🎁变回灰色
  - 若请求超时 → 使用同一个 `clientRequestId` 自动重试 1 次；若仍失败，再提示用户稍后重试
- 若奖池为空 → 弹出极简提示："许愿池空了，先许个愿望吧！" + [＋ 许个新愿望] 按钮

#### 4.2.4 互动操作区机制
- **道具列表**：读取配置（饼干、水果、球、飞盘、肉肉），横向滚动排列。
- **可用状态**：当前能量 >= 道具消耗值时高亮；否则置灰，Toast "能量不足啦"。
- **防连点**：播放互动动画 1.5 秒，期间操作台加透明遮罩。

**道具卡片三态 UI**：

| 状态 | 视觉 | 交互 |
|------|------|------|
| 默认(可用) | 全彩，边框正常 | cursor: pointer |
| Hover(可用) | 卡片上浮2px+阴影加深 | — |
| 点击(可用) | scale(0.95)→(1.0)弹回 | 触发互动 |
| 置灰(不可用) | 灰度+透明60% | cursor: not-allowed, Toast提示 |

> ⚠️ **V4 核心简化**：

#### 4.6.1 两个入口，各司其职

用户**点击金色🎁** → 系统自动执行以下流程（无需中间界面）：

1. **校验网络**：若当前离线 → Toast "网络不可用，恢复后再抽奖"，流程终止。
2. **发起抽奖请求**：前端生成 `clientRequestId`，立即调用 `POST /wishes/draw`。
3. **校验奖池**：若 `wishes(status=0)` 为空 → 弹出极简提示卡：

5. **揭晓庆祝**：服务端返回中奖心愿后，心愿卡放大居中 + 全屏 Confetti 纸屑。
6. **宠物联动**：气泡 P1："哇！[心愿名称] 中啦！快和爸爸妈妈说吧！"
7. **结算**：`pending_draw = false` → 🎁渐变回灰色。
8. **结果展示卡**（停留在屏幕上直到用户关闭）：
   ```text
   ╭──────────────────────────────╮
   │        🎉 恭喜中奖！         │
   │                              │
   │        🍔 吃一次麦当劳       │
   │        稀有度: ⭐⭐ 中愿望    │

   ```

> 整个抽奖从🎁点击到结果展示，**没有中间界面、没有心愿列表、没有历史记录**，极简到底。
> 若请求超时，前端使用同一个 `clientRequestId` 自动重试 1 次；若服务端已完成首次抽奖，重复请求必须返回第一次结果。

## 5. 非功能性需求

### 5.1 性能需求
- **FCP**：< 1.5 秒。核心资源预加载。
- **TTI**：< 2.0 秒。
- **动画帧率**：≥ 60 FPS。使用 CSS3 硬件加速 (`transform`, `opacity`) 或 Lottie。
- **左右面板同步**：延迟 < 100ms（乐观更新）。

### 5.2 兼容性需求
- **核心场景**：iPad 横/竖屏 (768-1024px)；PC > 1024px。
- **兼容场景**：移动端 < 768px，降级上下布局。
- **浏览器**：Safari 14+, Chrome 90+, Edge/Firefox 最新3版。

### 5.3 数据与安全需求
- **离线支持**：IndexedDB 缓存 + 待同步队列 + 网络恢复重试；打卡、互动、心愿管理可离线，抽奖仅在线。
- **设备绑定**：`X-Device-Id` 为客户端生成并本地持久化的 UUID；服务端绑定 `users.device_id`，V1 暂不支持换机恢复或多设备登录。
- **防连点**：打卡 500ms 禁用 + 互动动画期间遮罩。
- **破坏性确认**：删除目标/心愿需二确认弹窗。
- **XSS 过滤**：所有输入框限制特殊脚本字符。
